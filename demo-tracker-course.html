What This App Does

Meet Demo Tracker — a tool for managing AI project demos

🎯 The Problem It Solves

Teams building AI projects need an organized way to track which demos are ready, what's in progress, and what hasn't started yet.

Demo Tracker is a web application that helps teams:

📋
Track Use Cases

List every AI project with its problem statement and solution

🎬
Monitor Demo Status

Know instantly if a demo is Ready, In Progress, or Not Started

💰
Calculate Costs

Break down each project's costs into components (API calls, infrastructure, etc.)

🔍
Search & Filter

Find specific projects by name, status, or industry

A Quick Tour

Imagine you're the lead of an AI projects team. Here's what you'd do in Demo Tracker:

1
Create a Use Case

Click "New Use Case" and fill in what problem you're solving and how your AI solves it

2
Track Progress

Mark the demo status as your team builds: Not Started → In Progress → Ready

3
Add Demo Details

When Ready, add the demo URL and access instructions so others can try it

4
Track Costs

Break down expenses (API calls, hosting, licenses) and see monthly/annual totals

💡
Why This Matters

Instead of scattered spreadsheets and Slack messages, teams have one central source of truth for all their AI project demos. Everyone sees the same status, costs, and access information.

Code Example: Type Definitions

The TypeScript types define what data looks like:

Code 
export type DemoStatus = 'Ready' | 'In Progress' | 'Not Started';

export interface UseCase {
  Id: number;
  Title: string;
  Industry: Industry[];
  DemoStatus: DemoStatus;
  DemoUrl?: string;
}
Plain English

DemoStatus is like a dropdown with exactly 3 choices: Ready, In Progress, or Not Started — no other values allowed.

A UseCase is a data object with several fields, like a form with specific boxes to fill in.

The Id is a unique number that identifies each use case.

Title is the name — just plain text.

Industry can hold multiple values (like both "Finance" and "Healthcare").

DemoStatus shows which stage the demo is in.

The question mark means DemoUrl is optional — only needed when status is "Ready".

Code Example: Fetching Data

This hook fetches use cases from the API:

Code 
useEffect(() => {
  async function fetchUseCases() {
    const response = await fetch('/api/nocodb/use-cases');
    const data = await response.json();
    setAllCases(data.list);
  }
  fetchUseCases();
}, []);
Plain English

When the component first loads, run this code automatically.

Define a function that will fetch the data.

Call the API endpoint /api/nocodb/use-cases and wait for the response — this is like sending a messenger to get information.

Convert the response from JSON format into actual JavaScript data we can use.

Store the list of use cases in the component's memory.

End of the function definition — now actually run it.

The empty brackets mean "run this only once" when the component first appears.

💡
Key Insight

The browser NEVER talks directly to NocoDB. It always goes through the API routes — this keeps your database credentials safe on the server.

Code Example: NocoDB Client

This is how the server talks to NocoDB — notice the credentials stay on the server:

Code 
// These live ONLY on the server
const BASE_URL = process.env.NOCODB_BASE_URL;
const API_TOKEN = process.env.NOCODB_API_TOKEN;

async function nocofetch(endpoint) {
  const response = await fetch(url, {
    headers: { 'xc-auth': API_TOKEN }
  });
  return await response.json();
}
Plain English

Environment variables are secret values stored in a config file — the browser never sees them.

BASE_URL is your NocoDB instance address (like a building's address).

API_TOKEN is like a key that proves you're allowed in — it stays server-side.

This function fetches data from NocoDB using the API.

We send the API token in the request header — like showing your ID at the door.

Get the response and convert it from JSON to JavaScript data.

The Clever Tricks

Smart patterns that make the app feel great

Pattern 1: Auto-Save with Debounce

When you edit cost components, changes save automatically after 2 seconds of inactivity — no "Save" button needed:

Code 
const debouncedSave = useMemo(() =>
  (value) => setTimeout(() => {
    saveToApi(value);
  }, 2000),
  []
);
Plain English

Create a "debounced" version of the save function — it waits before actually saving.

The debounce wraps the actual save logic and adds a 2-second delay.

Actually call the API to save the data — but only after waiting.

Wait 2000 milliseconds (2 seconds) before saving.

The empty brackets mean this function is created once when the component loads.

Pattern 2: Client-Side Filtering

Search and filters happen instantly in the browser — no waiting for the server:

Code 
const filteredCases = useMemo(() => {
  let results = [...allCases];
  
  if (filters.searchQuery) {
    results = results.filter(c =>
      c.Title.includes(query)
    );
  }
  return results;
}, [allCases, filters]);
Plain English

Create a "derived" list that's automatically updated when the source data changes.

Start with all cases as a fresh copy.

If there's a search query, narrow down the results.

Keep only cases where the title contains what the user typed.

Return the filtered list.

Recalculate whenever allCases or filters change — like a spreadsheet that updates automatically.

Pattern 3: Conditional Form Fields

Demo URL and access instructions only appear when status is "Ready" — this guides users to provide the right info at the right time:

1
Status = "Not Started"

No demo fields shown — doesn't apply yet

2
Status = "In Progress"

Still building — demo info not ready

3
Status = "Ready"

Demo URL and instructions fields appear — required to mark complete

💡
Why These Patterns Matter

These patterns are what make an app feel "smart" and "polished." When you're working with AI coding tools, mentioning these patterns will help them understand what you want.

When Things Go Wrong

How to debug when something breaks

Common Issues

🔴

"Failed to fetch"

Check if NocoDB is running and credentials are correct in .env.local

🟡

Data not saving

Check browser console for errors — look for red messages

🔵

Wrong data showing

Clear filters or refresh the page — might be filtered unexpectedly

🟢

Cost calculations wrong

Check that frequency is set correctly (Monthly vs One-time)

The Debugging Mindset

When something breaks, here's how engineers think:

1
Reproduce the bug

Can you make it happen again? If so, you can fix it.

2
Check the console

Press F12 → Console tab. Red error messages tell you exactly what went wrong.

3
Trace the flow

Where in the journey did it break? Browser → API → NocoDB?

4
Check the simplest things first

Is NocoDB running? Are credentials correct? Did you refresh?

💡
Talking to AI About Bugs

When something breaks, tell the AI: what you expected to happen, what actually happened, any error messages you saw, and what you've already tried. This gives them a head start on debugging.
